# powdr-wasm

[powdr-wasm](TODO) is a new [WebAssembly](TODO)-based zkVM that provides better
proving performance than [RISC-V](TODO).
It uses [Crush](TODO) to compile WebAssembly to a zkVM-friendly ISA, and
implements the ISA as frontend circuits in the [OpenVM](TODO) modular zkVM
framework.

# WebAssembly vs RISC-V

RISC-V has been chosen for many zkVMs because of its simplicity. 
However, its hardware-first design does not optimize for the cost model of
zkVMs.
WebAssembly was designed to be a portable format for programs, and has been
praised before as a better performing alternative to RISC-V in the [context of
zkVMs](https://hackmd.io/@leoalt/best-isa-ethereum) and
[blockchains](https://ethresear.ch/t/why-risc-v-is-not-a-good-choice-for-an-l1-delivery-isa-and-why-wasm-is-a-better-one/23491).

Notable features of WebAssembly include:
- Platform independent portable format
- High level structured information
- No stack in memory
- Standardized ISA and runtime

## Crush IR/ISA/compiler

[Crush](TODO) is an infinite-register based IR that provides compilation from
WebAssembly.
It flattens WASM's native stack and locals into infinite registers, enabling
data-flow optimizations and optimizing the compiled program.

RISC-V has 32 registers and only a few of them are used by compilers to perform
computation on a function's local values.
That forces register spilling into memory, where a virtual stack is implemented
by compilers.
This means that for many local operations in a high level program, the RISC-V
assembly code will need to fetch 2 operands from memory (virtual stack), apply
the computation, and write the result back to memory.

Crush solves this by using infinite registers relative to a frame pointer,
never spilling registers to memory, and therefore optimizing the 4 operations
RISC-V often needs for a local computation to a single one. 
As we see below in the experiments, this optimization alone provides massive
benefits compared to RISC-V zkVM proofs.

Furthermore, because Crush inherits high level information from WASM, there is
even more information that can be used to improve a zkVM's proof time, such as
frame sizes and registers' lifetimes.

## Upcoming

powdr-wasm has shown that WASM combined with a custom ISA/compiler like Crush
can be much stronger than RISC-V for zkVMs in terms of optimizations and proof
time.

The beginning is promising and exciting, leading to a lot more future work that
includes:

- Autoprecompiles
- zkVM circuit optimizations
- Crush compiler optimizations
- Crush formal verification
- OpenVM v2 support
- Hints

If you are interested in using Crush or powdr-wasm, or helping optimize them
even further, let us know!
